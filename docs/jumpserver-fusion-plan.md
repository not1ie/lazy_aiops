# JumpServer 能力融合实施蓝图

## 目标

在保持 Lazy Auto Ops 现有模块可用的前提下，分阶段引入堡垒机能力，最终覆盖「资产接入、授权控制、会话代理、操作审计、审批合规」闭环。

## 范围定义

### 本期要实现（建议）

- 统一资产接入：主机、K8s 集群、数据库、网络设备
- 统一会话入口：Web SSH / Web K8s Exec / DB 命令入口
- 审计基础：会话日志、命令审计、操作人追踪、检索导出
- 授权基础：基于角色和资产范围的访问控制

### 后续增强

- 双人审批、时间窗授权、临时授权
- 多因子认证（MFA）
- 高风险命令管控（黑白名单、拦截策略）
- 会话录像回放（命令时间轴 + 终端回放）

## 三期实施计划

## 一期（MVP，4~6 周）

目标：先打通“能管、能连、能审”。

交付项：

- 数据模型：
  - `jump_assets`（资产）
  - `jump_accounts`（账号/凭据映射）
  - `jump_permission_policies`（访问策略）
  - `jump_sessions`（会话）
  - `jump_commands`（命令审计）
- API：
  - 资产 CRUD、账号绑定、授权策略 CRUD
  - 发起会话、关闭会话、查询会话与命令日志
- 前端：
  - 新增「堡垒机」一级菜单（资产、授权、会话审计）
  - 主机与 K8s 页面接入统一会话入口
- 审计：
  - 命令级审计入库（用户、资产、时间、结果、来源IP）

验收标准：

- 可以按用户角色限制资产访问
- 可以从统一入口发起 SSH/K8s 会话
- 命令查询可按用户、资产、时间过滤

## 二期（增强，6~8 周）

目标：补齐“会话代理能力”和“可回放审计”。

交付项：

- 协议代理层：
  - SSH 代理（稳定断线重连、窗口尺寸同步）
  - K8s Exec 代理（namespace/pod/container 精细控制）
  - 数据库代理（MySQL/PostgreSQL 基础命令审计）
- 审计增强：
  - 会话事件流落盘（input/output/timecode）
  - 回放接口与前端回放页面
- 安全控制：
  - 敏感命令识别与告警（先告警后拦截可配置）

验收标准：

- 任意会话可回放
- 敏感命令命中后可生成告警
- 会话断链率和失败率可观测

## 三期（合规与企业特性，8~12 周）

目标：接近 JumpServer 企业使用场景。

交付项：

- 审批与授权：
  - 工单审批流集成（与 `workorder` 融合）
  - 临时授权、时间窗授权、自动回收
- 身份安全：
  - MFA、登录风险校验、IP 白名单
- 命令控制：
  - 黑名单阻断、白名单放行、命令模板
- 合规报表：
  - 审计报表导出、按月归档、保留策略

验收标准：

- 关键操作需要审批才可执行
- 支持按策略自动失效授权
- 满足审计抽查和报表导出

## 与现有模块融合点

- `rbac`：沿用现有用户/角色体系，新增堡垒机权限点
- `cmdb`：资产主数据继续在 CMDB，堡垒机模块只做访问与审计扩展
- `terminal` / `k8s` / `docker`：统一接入会话代理层，减少重复实现
- `alert` / `notify`：高风险命令与异常会话可直接进入告警与通知链路
- `workorder`：审批授权流程复用现有工单能力

## 推荐技术方案

- 代理层：Go WebSocket + PTY + 协议适配器（SSH/K8s/DB）
- 审计存储：
  - 元数据：MySQL / PostgreSQL
  - 回放流：对象存储或分卷文件
- 队列与异步任务：用于审计落盘、报表生成、归档

## 风险与控制

- 性能风险：高并发会话占用较高
  - 控制：连接池、限流、会话节点水平扩展
- 安全风险：凭据泄露、越权访问
  - 控制：凭据加密、最小权限、操作留痕、审批机制
- 复杂度风险：一次性实现过大
  - 控制：严格按三期拆分，先可用后增强

## 里程碑建议

- M1（第 2 周）：数据模型 + 核心 API 完成
- M2（第 4 周）：SSH/K8s 统一会话入口可用
- M3（第 6 周）：一期验收上线
- M4（第 10 周）：回放与代理增强完成
- M5（第 14 周）：审批/MFA/合规报表完成
